## 1. F6 — Text size (S/M/L/XL), clamped so text never exceeds its shape

### Root cause
No font-size control exists. Labels render at a hardcoded base size times the
zoom scale:
- Node label: `BoardRenderer.drawNode`, `fontSize: 13 * viewport.scale`
  (`BoardRenderer.swift:362`).
- Note text: `BoardRenderer.drawNote`, `fontSize: 12 * viewport.scale`
  (`BoardRenderer.swift:487`).

### Where the field lives — DECISION: on `Style`
Put `textSize` on **`Style`** (`DesignerModel/Style.swift`), not on
`NodeSemantic`/`Note`. Rationale:
- `Style` is already carried by nodes, notes, ink and edges, and is the type
  the style panel already seeds from / emits (`StylePanelModel.style`,
  `StyleControls`), so the whole panel plumbing is reused with no new channel.
- `Style` already has the tolerant-coding machinery (explicit `CodingKeys` +
  `init(from:)`/`encode(to:)` + `extra` bag). **`Note` does not** — it is a
  plain synthesized `Codable` with no `extra` bag (`Elements.swift:227-237`),
  so a field added there would not benefit from the tolerant pattern and any
  sibling unknown keys would still drop. Style is the correct home.
- Node labels and note text both flow through `Style`, satisfying "text notes
  AND shape/node labels" with one field.

### Model change (`DesignerModel/Style.swift`)

Add a small enum + a stored property. Follow the existing string-backed
open-enum idiom from `Semantics.swift` so unknown future values round-trip:

```swift
/// Presentational label size. A multiplier on the element's base font size.
/// nil == medium (1.0). Kept OFF the agent wire format (see WireBoard note).
public enum TextSize: String, Codable, Sendable, CaseIterable {
    case small, medium, large, xl

    public var multiplier: Double {
        switch self {
        case .small:  return 0.8
        case .medium: return 1.0
        case .large:  return 1.35
        case .xl:     return 1.8
        }
    }
}
```

In `struct Style`:
- Add `public var textSize: TextSize?` (after `image`).
- Add to `init(...)` (param `textSize: TextSize? = nil`, assign it).
- `CodingKeys` (line 47): add `textSize` to the case list. (`CaseIterable` +
  `knownKeys` already excludes it from the `extra` bag automatically.)
- `init(from:)` (line 51): `textSize = try container.decodeIfPresent(TextSize.self, forKey: .textSize)`.
- `encode(to:)` (line 61): `try container.encodeIfPresent(textSize, forKey: .textSize)`.
- Add a convenience: `public var effectiveTextMultiplier: Double { (textSize ?? .medium).multiplier }`.

### Wire-format decision — DECISION: NOT on the wire (recommended)
Do **not** add `textSize` to `WireBoard.WireNode` (`DesignerInterop/WireBoard.swift:36`).
Justify per CLAUDE.md: the wire format is deliberately lossy on presentational
fields — `image`, node `extra` (board links), boundaries, ink never cross it.
`textSize` is pure presentation. Because proposals parse **anchored to the
current board** and matched nodes inherit `Style` fields the agent leaves unset
(`LLMInterchange` style inheritance), a matched node keeps its `textSize`
automatically. Adding it to the wire would let an agent silently reset user
typography on every proposal — exactly the "can't wipe what they can't see"
guarantee. Leave it off; state this in a code comment on the `TextSize` enum.

### Rendering hook + clamp (`BoardRenderer.swift`)
Both `drawNode` and `drawNote` must scale the base font by the element's
multiplier and then **clamp so the rendered text box never exceeds the frame**.

Add one helper to `BoardRenderer`:

```swift
/// Font size for a label: base × the element's textSize multiplier × zoom,
/// then clamped so a single line of `text` fits inside `frameView` (both the
/// view-space frame). Prevents XL text from spilling out of a small shape.
private func clampedLabelFontSize(
    base: CGFloat, multiplier: Double, text: String,
    frameView: CGRect, viewport: CanvasViewport
) -> CGFloat {
    let requested = base * CGFloat(multiplier) * CGFloat(viewport.scale)
    guard !text.isEmpty else { return requested }
    // Measure at the requested size, shrink by the limiting axis ratio.
    let measured = attributedString(text, fontSize: requested, color: .black).size()
    let maxW = max(frameView.width - 8 * viewport.scale, 1)
    let maxH = max(frameView.height - 8 * viewport.scale, 1)
    let widthRatio  = measured.width  > maxW ? maxW / measured.width  : 1
    let heightRatio = measured.height > maxH ? maxH / measured.height : 1
    let ratio = min(widthRatio, heightRatio)
    // Never shrink below the readable floor the renderer already respects.
    return max(requested * ratio, min(requested, 9))
}
```

Note `attributedString` (line 1327) is `private` and already buckets sizes for
its cache — reusing it here is free. `drawText(_:...centeredIn:)` (line 1251)
already wraps/truncates for width; the clamp handles the font-height overflow
that wrapping can't fix (XL in a short frame).

- `drawNode` (line 360-366): replace `fontSize: 13 * viewport.scale` with
  `fontSize: clampedLabelFontSize(base: 13, multiplier: node.style.effectiveTextMultiplier, text: node.semantic.name, frameView: textRect, viewport: viewport)`.
  Use `textRect` (already computed, accounts for image label strip / cylinder
  lid), not `rect`.
- `drawNote` (line 484-491): same, `base: 12`,
  `multiplier: note.style.effectiveTextMultiplier`, `text: note.text`,
  `frameView: rect`.

Perf: the extra measurement is one cached `attributedString(...).size()` per
visible label per frame at near zoom only (`viewport.scale >= textVisibilityScale`,
where labels already draw). The far-zoom batch path draws no text, so it is
untouched. The text cache (line 1330-1344) already survives continuous zoom via
size bucketing; the clamp feeds it bucketed sizes, so no cache thrash.

## 3. B2 — Connector caption flicker during zoom

### Root cause
`BoardRenderer.beginCaptionPass()` (`BoardRenderer.swift:60`) rebuilds a fresh
greedy stateful `EdgeGeometry.CaptionPlacer` every frame. In the near-zoom draw
loop (`CanvasView.swift:552`) each visible edge calls `captionPlacer.place(...)`
(`EdgeGeometry.swift:315`) with `pillSize` derived from a **view-points**
measurement divided by scale (`BoardRenderer.swift:660-662`): text is measured
at `12 * viewport.scale`, then `/ viewport.scale`, so font rounding/bucketing
(line 1331) makes the *world* pill size wobble frame-to-frame as scale changes.
Combined with culling changing the obstacle set and placement order mid-zoom,
the greedy first-fit picks a different slot each frame → the pill jumps.

### Approach — solve once per settled viewport, reuse while in flight
Two coordinated changes: (a) make pill size scale-independent so a solve is
stable, and (b) only re-solve when the viewport is settled; otherwise reuse the
cached placement.

#### (a) Scale-independent pill size for placement (`BoardRenderer.swift`)
`captionContent` (line 675) measures at `12 * viewport.scale` / `10 * viewport.scale`.
Add a placement-only measurement at a FIXED reference scale (1.0) so the world
pill size does not depend on zoom:

```swift
/// World-space pill size measured at scale 1 — stable across zoom, used only
/// for placement (rendering still measures at the live zoom).
func worldCaptionPillSize(for edge: Edge) -> Size? {
    guard let content = captionContent(for: edge, viewport: CanvasViewport(scale: 1))
    else { return nil }
    return Size(width: Double(content.pillSize.width), height: Double(content.pillSize.height))
}
```

#### (b) Persist the resolved caption center; re-solve only when settled
Move the decision of WHICH placement each edge gets out of the per-frame draw
loop and into a cache keyed by edge id, recomputed only when the viewport is
settled.

On `BoardRenderer`:
- Add `private(set) var captionCenters: [ElementID: Point] = [:]` (resolved
  world centers).
- Change `beginCaptionPass()` → `beginCaptionPass(resolve: Bool)`. Store the
  flag; when `resolve` is true, reset `captionPlacer` and clear a scratch set of
  freshly-placed ids.
- Change `drawEdge`/`drawEdgeContent` to take `edgeID: ElementID`. In the
  caption block (line 654-666):
  - if `resolve` (settled): compute `center` via `captionPlacer.place(preferred:
    captionFraction, route:, pillSize: worldCaptionPillSize(for: edge)!, obstacles:)`
    and store `captionCenters[edgeID] = center`.
  - else (in flight): `center = captionCenters[edgeID] ?? route.point(atFraction: captionFraction)`.
  - draw the caption at `viewport.toView(center)` as today.

On `CanvasView`:
- Add `private var captionsDirty = true` and `private var lastCaptionViewport: CanvasViewport?`.
- Add `private var viewportSettleWork: DispatchWorkItem?`.
- `board` didSet (line 18): set `captionsDirty = true` (routes changed → caption
  layout must be recomputed).
- `viewport` didSet (line 148): schedule a settle:
  ```swift
  viewportSettleWork?.cancel()
  let work = DispatchWorkItem { [weak self] in
      self?.captionsDirty = true
      self?.needsDisplay = true
  }
  viewportSettleWork = work
  DispatchQueue.main.asyncAfter(deadline: .now() + 0.10, execute: work)
  ```
- `animateViewport` completion (line 1081-1088): set `captionsDirty = true`.
- In `draw`, before the near-zoom loop, decide:
  ```swift
  let resolveCaptions = captionsDirty && cameraAnimation == nil
  renderer.beginCaptionPass(resolve: resolveCaptions)
  ```
  and after the drawables loop, `if resolveCaptions { captionsDirty = false; lastCaptionViewport = viewport }`.
- Pass `edgeID: element.id` into the two `renderer.drawEdge(...)` calls
  (lines 561 and 470/482 fast-path — the fast path uses `simplified: true` and
  draws no caption, so it can pass the id but it is ignored there).

Effect: during a pinch/scroll/animated dive the pill stays pinned to its last
solved world center (it simply scales with the view, no re-solve, no jump). ~100 ms
after the gesture stops, one solve runs and the layout settles into its final
collision-avoided slots.

### Perf
The per-frame `place()` call (a greedy scan of up to 11×5 candidate rects with
obstacle queries per visible edge) is removed from the hot path entirely except
on the single settle frame. During steady state and during zoom, caption draw
is a dictionary lookup + one `toView`. Net win. `captionCenters` is bounded by
edge count and rebuilt on the settle frame; stale entries for deleted edges are
harmless (never read) but can be pruned in `board` didSet if desired.

### Test
- **Headless (`DesignerCanvasTests/RoutingVisualCheck.swift` or a new
  `CaptionPlacementTests`)**: build a board with two parallel labelled edges.
  Call the placement solve at scale 1.0, capture `captionCenters`. Call it again
  at scale 3.0 (same viewport origin) — assert the WORLD centers are within a
  tiny epsilon of the scale-1 centers (proves scale-independence). This is the
  regression that pins the flicker.
- **UITest step `step32CaptionStable()`**: create two connected labelled nodes,
  zoom via `canvasView.viewport = CanvasViewport(scale: k)` across k = 1, 1.5,
  2.5 WITHOUT pumping the settle delay, capturing `renderer.captionCenters`
  (expose read-only for tests) each time; assert the cached world center does
  NOT change between zooms (in-flight reuse). Then wait past the 0.10 s settle
  (spin the runloop) and assert it re-solves at most once.

## 4. B3 — Style panel covers a shape created near the left edge

### Root cause
The left style/inspector column is installed at `leading + 16`, width `236`
(`CanvasViewController.swift:80`, `StylePanel.frame(width: 236)` at
`StylePanel.swift:109`), so it overlays view-x ≈ 16…252 whenever visible. An
element created there (double-click note, dragged shape, block) is hidden
behind the panel, and the panel "swallows clicks" for anything under it
(CLAUDE.md warning: keep UI-test scenes clear of view-x ≲ 260).

### Approach — auto-pan after creation so the element clears the panel band
After an insert whose view rect underlaps the panel band while the panel is (or
will be) visible, pan the viewport horizontally so the element's left edge sits
just right of the panel.

Add one helper to `CanvasView`:

```swift
/// If `frame`'s on-screen rect underlaps the left style-panel band while a
/// panel is showing, pan so it clears the band. View x-band ≈ 0…268
/// (leading 16 + width 236 + 16 gap). No-op when the panel is hidden.
private func nudgeClearOfLeftPanel(_ frame: Rect) {
    guard leftPanelIsVisible else { return }          // see wiring below
    let band: CGFloat = 268
    let viewRect = viewport.toView(frame)
    guard viewRect.minX < band else { return }
    let dx = band + 12 - viewRect.minX                 // pan content right by dx
    viewport.pan(viewDeltaX: dx, viewDeltaY: 0)        // origin.x -= dx/scale
}
```

`pan(viewDeltaX:)` shifts every world point by `+dx` in view space
(`CanvasViewport.swift:51`: `origin.x -= dx/scale`; a fixed world point's new
view-x is `oldViewX + dx`). So `dx = band + margin − viewRect.minX` moves the
element's left edge exactly to `band + margin`.

Wiring the "panel visible" signal into `CanvasView` (which lives in
DesignerCanvas and cannot see the controller): add
`public var leftPanelIsVisible: Bool = false` on `CanvasView`, and have
`CanvasViewController.refreshStylePanel` keep it in sync at the end:
`canvasView.leftPanelIsVisible = stylePanelModel.isVisible || inspectorModel.visible`.

Call `nudgeClearOfLeftPanel(frame)` at the tail of the three creators, AFTER the
`perform` + `selection` are set (so the just-created element is on-screen and
its label editor, which reads the frame, positions correctly):
- `insertTextNote(at:)` (`CanvasView.swift:2003`) — after `beginLabelEdit`.
- `insertBlock(...)` (line 2403) — after `beginLabelEdit`.
- `insertShape(worldRect:...)` (line 2438) — after `selection = [element.id]`.

Because the label editor is created from the world frame and follows the
viewport, panning after `beginLabelEdit` is fine; if the editor caches a view
rect, re-call its reposition (there is a viewport-didSet → `needsDisplay`; verify
the editor tracks — `B1` fix already made it scale with the zoomed node, so it
reads the live frame).

Alternative (rejected): offsetting the *placement* world point instead of
panning. Rejected because double-click-to-create should land where the user
clicked; panning keeps the element under the cursor's world point and just
brings the camera along, which is less surprising.

### Perf / undo
Pure view-state change (viewport), no board mutation, no undo entry, no
per-frame cost.

### Test
- **UITest step `step33ShapeClearsPanel()`**: ensure the style panel is visible
  (arm the shape tool). Place a shape whose world rect maps to view-x < 100 with
  the panel open (e.g. set a viewport so world (0,0) is at view-x ≈ 40). After
  creation, assert `canvasView.viewport.toView(newNode.frame).minX >= 268`
  (element cleared the band). Assert the board still has exactly one new node
  (pan did not create/lose anything). Note: this is the one step allowed to
  START under the band — it verifies the auto-pan, then everything else is
  clear.

## 5. B10 — "Weird edges" routing artifacts when dragging nodes

### Root cause (established)
1. **Drag re-resolve is not the settled path.** During a live drag only
   `dragAffectedEdges` re-resolve per frame (`CanvasView.swift:406-425`) via
   `EdgeGeometry.route(for:frames:parallelOffset:anchorOffsets:)` using the
   **stale** `parallelOffsetCache`/`anchorSpreadCache` (computed at the last
   `board` didSet) and passing **no `obstacles:`** argument. The settled routes
   (`SpatialIndex.resolveRoutes`, `SpatialIndex.swift:60-77`) DO pass
   `obstacles: nodeObstacleQuery(for:board)`. So during a drag an edge does not
   avoid blocks and uses stale spread, then "pops" to a different route on drop.
2. **`autoSide` flips mid-drag.** As the dragged node moves, `targetPoint`
   changes and `autoSide` (`EdgeGeometry.swift:582`) picks a different border →
   the arrowhead jumps sides while dragging.
3. **Residuals inside `route()`**:
   - **Hook**: the detour re-anchor (lines 164-176) only accepts the re-anchored
     endpoint when `sideFaces` passes; otherwise it keeps the straight-line
     anchor while the smoothed curve arrives from the detour side — the curve
     meets a border point that faces the wrong way (a hook/kink).
   - **Cross**: a cluster whose detour would exceed 220 pt is skipped and
     crossed (`avoidanceWaypoints`, line 294).
   - **Straight-through fallback**: if the mirrored detour still intersects
     blockers, `route()` returns the straight `points` through them (line 185).
   - **Spread mis-co-planning**: `anchorSpread` (line 445) only groups pure-auto
     anchors (`case .element(nodeID, nil, nil)`, line 467); a re-anchored
     endpoint (given an explicit offset) is not co-planned, so siblings still
     converge on the same border point.

### Fixes (prioritized)

**(a) During drag, route affected edges through the same path as
`resolveRoutes` — with live obstacles.** In `CanvasView.draw`, build a
transient obstacle query once per frame from `overrideFrames`, and pass it to
the drag re-resolve. Add to `SpatialIndex`:

```swift
/// Obstacle query over live/overridden node frames (drag-time analogue of
/// nodeObstacleQuery, which reads only committed frames).
public static func nodeObstacleQuery(
    for board: Board, overrides: [ElementID: Rect]
) -> (Rect) -> [Rect] {
    var index = SpatialIndex()
    var frames: [ElementID: Rect] = [:]
    for element in board.elements.values {
        guard element.node != nil else { continue }
        let frame = overrides[element.id] ?? element.node!.frame
        frames[element.id] = frame
        index.insert(element.id, bounds: frame)
    }
    return { rect in index.query(rect).compactMap { frames[$0] } }
}
```

In `CanvasView.draw` (near line 400), when `overrideFrames != nil` build
`let dragObstacles = SpatialIndex.nodeObstacleQuery(for: board, overrides: transientFrames)`
once, and in the `routeFor` closure's drag branch (line 420-425) pass
`obstacles: dragObstacles`. This makes drag routes avoid blocks and match the
settled route the drop will produce (kills the "pop").

Perf: builds one spatial index over nodes per frame **only while a drag is in
flight** (`transientFrames` non-empty). Dense boards: O(nodes) build; acceptable
for an active gesture (not steady state). If measured too costly on 4k-node
boards, cache the index per drag and only re-insert the moved ids each frame —
note this as a follow-up, not required for correctness.

**(b) Re-validate the smoothed spline against the target's own border and
prefer the facing side** (`EdgeGeometry.route`). After the detour block
finalizes `points` (line 188), and generally at the end of `route()`, add an
arrival-correction pass:

```swift
// Arrival correction: the final polyline must reach each element endpoint on a
// border that FACES the approach, and must not dive through the node interior.
points = correctedArrival(points, edge: edge, frames: frames,
                          fromOffset: anchorOffsets?.from, toOffset: anchorOffsets?.to)
```

`correctedArrival` (new private static): for each element endpoint, take the
approach direction from the neighbouring sample (`points[1]` for from,
`points[count-2]` for to), compute `autoSide(from: frame, toward: approachPoint)`,
and if that differs from the side the endpoint currently sits on, re-`resolve`
the endpoint on the facing side and replace the endpoint sample; re-`smoothed`
the last/first segment so the curve doesn't kink. This replaces the
"keep the straight-line anchor when sideFaces fails" hook with "always land on
the side that faces where the line actually comes from."

**(c) Co-plan spread with re-anchor.** Make `anchorSpread` group by the side the
edge will ACTUALLY arrive on after detour/correction, not only pure-auto
anchors. Two-pass in `resolveRoutes`:
1. First pass: route all edges with `spread = anchorSpread(in: board)` (as
   today) and record each edge's resolved arrival `side` per endpoint (return it
   from `route` via a lightweight out-struct, or recompute `autoSide` on the
   final polyline).
2. Recompute spread grouping on the OBSERVED sides (including re-anchored edges),
   then route again with the corrected spread.

`resolveRoutes` runs on `board` didSet only (not per frame), so a second pass is
affordable; gate it to run only when the first pass produced any detour/
re-anchor (skip on boards with no obstacle interactions). Mark (c) as the
deepest change — ship (a)+(b) first; (b) alone removes the visible hook and
wrong-border arrival, (a) removes the drag-vs-drop pop, (c) is the polish that
stops re-anchored siblings from converging.

### Undo / wire
No model or persisted change — all geometry. No undo impact (routes are derived,
not stored). No wire impact.

### Test
- **Headless (`DesignerModelTests/EdgeGeometryTests.swift`)**: the load-bearing
  regressions.
  - Build three node frames in a row (A left, C middle blocker, B right) and an
    edge A→B. `route(for:frames:obstacles:)` must (i) not sample inside C's
    frame (reuse the private `routeIntersectsBlockers` logic — expose an
    `internal` test shim `EdgeGeometry.routeHitsBlocker(_:obstacles:)`), and
    (ii) arrive at B on a side whose `autoSide(from: Bframe, toward: penultimate)`
    equals the arrival side (facing).
  - Move C so the straight line barely clips it; assert no straight-through
    fallback (the returned polyline is not exactly `[Aanchor, Banchor]`).
  - Parallel siblings A→B ×2 with a blocker: assert their arrival border points
    on B differ (spread co-planned) — pins fix (c).
- **UITest step `step34WeirdEdgeRouting()`** (real mouse events on a dense
  synthetic graph): build ~6 nodes in two rows with ~8 connectors (a synthetic
  stand-in for the Astelia layout, since that board is not committed). Drag one
  interior node BETWEEN two connected nodes with real
  `send(.leftMouseDragged, ...)` steps (model on `step3DragBlock`/`step7`), pump,
  then for every edge anchored to a moved node assert, on the committed board's
  `SpatialIndex.resolveRoutes` output: (1) the arrival point lies on the target

## 6. F4 — Broken board links: distinct badge + hover popover

### Root cause / constraints
Links live in `node.semantic.extra["linkedBoard"]` as a `BoardID`
(`NodeLink.swift`). Resolution to a file is `BoardCatalog.url(forBoardID:)`
(`BoardCatalog.swift:145`) which calls `entries()` — a **filesystem scan that
reads every board.json**. This must NEVER run per frame at 120 Hz. The badge is
drawn by `BoardRenderer.drawLinkBadge` (`BoardRenderer.swift:1152`) at
`CanvasView.linkBadgeRect` (`CanvasView.swift:1095`); `drawLinkBadges`
(line 1113) iterates only linked nodes. There is **no hover / tracking-area
infrastructure** on the canvas today.

### Approach

#### (a) Cheap cached validity — resolve off the hot path
Add to `CanvasView`:
- `private var brokenLinkIDs: Set<ElementID> = []`
- `public var resolveLinkValidity: ((BoardID) -> Bool)?` — injected by the
  controller so DesignerCanvas need not depend on `BoardCatalog`.
- `public func refreshBrokenLinks()`:
  ```swift
  guard let resolve = resolveLinkValidity else { brokenLinkIDs = []; return }
  var broken: Set<ElementID> = []
  var cache: [BoardID: Bool] = [:]           // dedupe repeated board ids
  for element in zOrderedElements {
      guard let id = element.node?.semantic.linkedBoardID else { continue }
      let ok = cache[id] ?? { let v = resolve(id); cache[id] = v; return v }()
      if !ok { broken.insert(element.id) }
  }
  if broken != brokenLinkIDs { brokenLinkIDs = broken; needsDisplay = true }
  ```

Call `refreshBrokenLinks()` from:
- `board` didSet tail (line 44) — but guard cost: it triggers `entries()`. To
  keep `board` didSet cheap, do NOT call synchronously there; instead set a flag
  and dispatch async: `DispatchQueue.main.async { [weak self] in self?.refreshBrokenLinks() }`
  so the scan runs after the frame, once, coalesced.
- Controller wiring: in `CanvasViewController.installLinkedBoards`
  (`LinkedBoards.swift:139`) set
  `canvasView.resolveLinkValidity = { BoardCatalog.url(forBoardID: $0) != nil }`
  and call `canvasView.refreshBrokenLinks()`.
- On window focus (catalog may have changed in another window): observe
  `NSWindow.didBecomeKeyNotification` in the controller → `canvasView.refreshBrokenLinks()`.
- After the app trashes/creates a board (`BoardCatalog.trash`,
  `createBoardWithLink`): post a `Notification.Name("DesignerCatalogChanged")`
  and have the controller re-run `refreshBrokenLinks()`.

#### (b) Distinct broken badge (`BoardRenderer.swift`)
Give `drawLinkBadge` a `broken` flag:
```swift
func drawLinkBadge(in rect: CGRect, broken: Bool = false, context: CGContext) {
    context.saveGState()
    context.setFillColor((broken ? Graphite.warning /* amber, add if absent */
                                  : Graphite.accent).cgColor)
    ... fill ellipse ...
    if broken {
        // Draw a "!" glyph instead of the ↗ arrow.
        drawText("!", fontSize: rect.height * 0.7, color: .white,
                 centeredIn: rect, context: context)
    } else {
        ... existing ↗ arrow strokes ...
    }
    context.restoreGState()
}
```
If `Graphite` has no amber, add one (e.g. `#E8943A`, already in the swatch
palette). `drawLinkBadges` (`CanvasView.swift:1118`) passes
`broken: brokenLinkIDs.contains(element.id)`.

#### (c) Hover popover + broken activation
Add hover infrastructure to `CanvasView`:
- Override `updateTrackingAreas()` to install one `NSTrackingArea` covering
  `bounds` with options `[.mouseMoved, .activeInKeyWindow, .inVisibleRect]`
  (remove/re-add the old one each call). This is the standard pattern; the
  canvas has none today.
- Override `mouseMoved(with:)`: hit-test the badge rects of broken links only
  (cheap — iterate `brokenLinkIDs`):
  ```swift
  let p = convert(event.locationInWindow, from: nil)
  let hit = brokenLinkIDs.first { id in
      guard let f = board.elements[id]?.node?.frame else { return false }
      return linkBadgeRect(forNodeFrame: f).insetBy(dx: -3, dy: -3).contains(p)
  }
  if hit != hoveredBrokenLink {
      hoveredBrokenLink = hit
      brokenLinkHoverChanged?(hit, hit.flatMap { board.elements[$0]?.node?.frame }
          .map { linkBadgeRect(forNodeFrame: $0) } ?? .zero)
  }
  ```
- Add `public var brokenLinkHoverChanged: ((ElementID?, CGRect) -> Void)?` and
  `public var brokenLinkActivated: ((ElementID) -> Void)?`; the controller owns
  the popover UI (SwiftUI content lives in the Designer target).

Controller (`LinkedBoards.swift`): implement a small `NSPopover` with a SwiftUI
body ("This block links to a board that isn't in your catalog anymore." + a
"Re-link…" button → `presentLinkPicker(for:)`). Show it on
`brokenLinkHoverChanged(id, rect)` anchored to `rect` in `canvasView`; dismiss on
`nil`.

Activation: in `CanvasView.mouseDown` badge branches (lines 1529 and 1560) and
in the linked-board read-only branch, before calling `linkActivated?`, check
`if brokenLinkIDs.contains(linked) { brokenLinkActivated?(linked); return }`.
Also short-circuit the model-level entry `enterLinkedBoard`
(`LinkedBoards.swift:270`) already alerts on a missing URL — reuse that alert
text, but prefer the popover explanation for the badge path.

### 120 Hz constraint — explicit
- The filesystem scan (`entries()`) runs ONLY on: async-after-board-change,
  window focus, catalog-change notification. Never per frame, never in
  `drawLinkBadges`.
- Per-frame cost is a `Set.contains` per linked node (already the only nodes
  iterated) — O(linked nodes), trivial.
- Hover hit-test runs only on `mouseMoved` and only over `brokenLinkIDs`
  (usually empty/small).
- Broken badges are drawn in the same overlay pass as normal badges
  (`drawLinkBadges` is called after the main passes, not in the far-zoom batch),
  so no fast-path impact.

### Test
- **Headless**: `BoardCatalog.url(forBoardID:)` returning nil for a random
  BoardID is already implied; add a `CanvasView`-level unit isn't possible
  headless (AppKit). Instead test the resolution closure shape: a
  `DesignerModel`/persistence test that a node with a bogus `linkedBoardID`
  reports `linkedBoardID != nil` but resolves to no URL.
- **UITest step `step35BrokenLink()`**: create a node, set
  `node.semantic.linkedBoardID = BoardID(UUID())` (a board that doesn't exist)
  via `document.perform(.replaceElement(...))`, call
  `canvasView.refreshBrokenLinks()`, pump, assert
  `canvasView.brokenLinkIDs.contains(id)`. Synthesize a `mouseMoved` over the
  badge rect (`send(.mouseMoved, at: badgeCenter, ...)` — extend the `send`
  helper to allow mouseMoved) and assert `brokenLinkHoverChanged` fired with the
  id. Synthesize a click on the badge and assert `brokenLinkActivated` fired
  (not `linkActivated`, i.e. no navigation). Render a PNG and assert the badge
  region contains amber pixels (broken style), contrasting a control node with a
  valid link (accent). Register + extend PASS line.

## 7. F8 — Z-order clarity vs current layer (design proposal to iterate)

**This is an explicit design proposal to show the user before polishing**
("let's design this and see if I like the results"). Ship the first cut, get a
reaction, then refine. Keep it low-risk and reversible.

### Concept
Z-order is a single board-global `sortKey` (`Element.sortKey`); layers are an
orthogonal multi-membership/visibility concern (`Element.layerIDs`). Users
confuse "send to back" (global z) with "which layer am I on." First cut surfaces
both near the existing To Back / To Front buttons in the style panel:
1. A compact **"Layer: <name>"** chip (the selection's layer; "Mixed" if the
   selection spans layers).
2. A **z-position readout** counting only elements that SHARE the selection's
   layer(s): e.g. "3rd from front of 12 on Data".
3. **Forward / Backward one-step** buttons (in addition to To Back / To Front),
   disabled at the extremes.

### Files & wiring
- **`DesignerCanvas/CanvasView.swift`** — add one-step reorder + a z-position
  query:
  - `public func stepSelectionForward()` / `stepSelectionBackward()`: like
    `reorderSelection` (line 2476) but instead of jumping to `topSortKey` /
    bottom, move the element ONE position among elements sharing at least one of
    its `layerIDs`, sorted by `(sortKey, id)`. Compute the neighbour above/below
    in that filtered, sorted list and set `sortKey = SortKey.between(lower, upper)`
    accordingly; commit via the delegate as one `.batch` "Bring Forward" /
    "Send Backward". No-op (and disable) at the extreme.
  - `public func zPosition(of id: ElementID) -> (rank: Int, total: Int)?`:
    filter `board.elementsInZOrder` to those sharing a layer with the element,
    return its 1-based rank from front and the count. Used by the controller to
    format the readout.
- **`Designer/StylePanel.swift`** — extend `StylePanelModel` with published
  `layerChipText: String?`, `zPositionText: String?`, `canStepForward: Bool`,
  `canStepBackward: Bool`; extend `StylePanelActions` with `stepForward` /
  `stepBackward`. In `StyleControls`, in the z-order block (line 214), when
  `mode == .selection || .image`, render the chip + readout above the button row
  and add Forward/Backward buttons beside To Back/To Front, `.disabled(!canStep…)`.
- **`Designer/CanvasViewController.swift`** — in `refreshStylePanel` (line 243),
  when a single node/note/image is selected, populate `layerChipText`
  (join layer names, "Mixed" if >1), `zPositionText` (from
  `canvasView.zPosition(of:)`, "Nth from front of M on <layer>"), and the
  can-step flags. Wire `stepForward`/`stepBackward` actions in
  `installStylePanel` (line 184) to the new `CanvasView` methods.
- **`Designer/LayersPanel.swift`** — optional in the first cut: emphasize the
  active layer row (bold / accent) so the panel and the chip agree. Note as a
  follow-up if it complicates the diff.

### Model / undo / wire / perf
- No model or persisted change — `sortKey` already exists; one-step reorder is
  the same `SortKey.between` machinery as `reorderSelection`, one undo step.
- The z-position query runs on selection change (`refreshStylePanel`), not per
  frame; it is O(elements) over the cached `elementsInZOrder`. Fine.
- No wire impact.

### Test
- **Headless (`DesignerModelTests/SortKeyTests.swift` or `LayerCommandsTests`)**:
  three elements on the same layer with known sortKeys; assert
  `stepSelectionForward` on the middle one swaps only its rank by one and leaves
  the others; assert a step at the front extreme is a no-op. Assert the
  z-position count excludes elements NOT sharing a layer.
- **UITest step `step36ZOrderClarity()`**: create three overlapping nodes on one
  layer, select the middle, assert `controller.stylePanelModel.zPositionText`
  reads "2nd" of 3, `canStepForward == true`, `canStepBackward == true`. Call
  `canvasView.stepSelectionForward()`, pump, assert the readout updates to "1st"
  and `canStepForward == false`. Because this is a proposal, keep the assertions
  on the DATA (`zPosition`, sortKey ordering), not pixel layout, so the visual
  can be iterated without breaking the test. Register + extend PASS line.
